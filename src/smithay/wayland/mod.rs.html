3//! Protocol-related utilities
4//!
5//! This module contains several handlers to manage the Wayland protocol
6//! and the clients.
7//!
8//! ## General structure
9//!
10//! Most utilities provided in this module work in the same way:
11//!
12//! - A module specific `*State` struct will take the wayland display as argument and
13//!   insert one or more globals into it through its constructor.
14//! - The module-`State` will have to be stored inside your global compositor state.
15//!   (The same type you parametrized [`wayland_server::Display`] over.)
16//! - You need to implement a module-specific `*Handler`-trait for your compositor state.
17//!   This implementation will be called when wayland events require custom handling.
18//! - Call the matching `delegate_*!` macro from smithay on your state to implement
19//!   some required `wayland_server` traits.
20//! - If you want to remove a previously inserted global, just drop the `*State`.
21//!
22//! ## Provided helpers
23//!
24//! ### Core functionality
25//!
26//! The most fundamental module is the [`compositor`] module, which provides the necessary
27//! logic to handle the fundamental component by which clients build their windows: surfaces.
28//! Following this, the [`shell`] module contains the logic allowing clients to use their
29//! surface to build concrete windows with the usual interactions. Different kind of shells
30//! exist, but in general you will want to support at least the [`xdg`](shell::xdg) variant,
31//! which is the standard used by most applications.
32//!
33//! Then, the [`seat`] module contains logic related to input handling. These helpers are used
34//! to forward input (such as pointer action or keystrokes) to clients, and manage the input
35//! focus of clients. Tightly coupled with it is the [`selection`] module, which handles
36//! cross-client interactions such as accessing the clipboard, or drag'n'drop actions.
37//!
38//! The [`shm`] module provides the necessary logic for client to provide buffers defining the
39//! contents of their windows using shared memory. This is the main mechanism used by clients
40//! that are not hardware accelerated. As a complement, the [`dmabuf`] module provides support
41//! hardware-accelerated clients; it is tightly linked to the
42//! [`backend::allocator`](crate::backend::allocator) module.
43//!
44//! The [`output`] module helps forwarding to clients information about the display monitors that
45//! are available. This notably plays a key role in HiDPI handling, and more generally notifying
46//! clients about whether they are currently visible or not (allowing them to stop drawing if they
47//! are not, for example).
48//!
